# Habitica Party Automation Bot

This document provides a comprehensive overview of how the Habitica Party Automation Bot works, what happens in the backend, how the code is structured, and exactly what changes it makes to your Habitica account and party.

## 1. Project Overview
This project is a Node.js-based backend application designed to automate common administrative tasks for a Habitica Party. It runs continuously as a server, listening for events (via Webhooks) and running scheduled tasks (via Cron jobs) to manage party activities automatically.

## 2. Backend Architecture & Implementation

The backend is built using Node.js and the Express framework. It relies on the official Habitica API (v3) to interact with your party data. 

Here is a breakdown of the key JavaScript files and what they do:

### `server.js` (The Entry Point)
This is the main file that starts the Express web server. 
- **What it does:** It sets up an HTTP server listening on a specified port. It exposes endpoints like `/health` (to check if the server is running) and `/webhook/chat` (to receive messages from Habitica). 
- **Initialization:** When started, it checks for your `.env` credentials, starts listening to webhooks, and calls `initializeScheduler()` to start all background tasks.

### `habiticaClient.js` (The API Bridge)
This file handles all communication between the bot and Habitica's servers.
- **What it does:** It uses the `axios` library to send HTTP requests to the Habitica API. It uses your `HABITICA_USER_ID` and `HABITICA_API_KEY` to authenticate.
- **Rate Limiting:** It includes a built-in rate limiter to ensure the bot doesn't send more than 30 requests per minute, preventing your account from being temporarily banned by Habitica.
- **Actions:** It contains methods to get party members, get party info, send chat messages, update the party description, and force-start quests.

### `scheduler.js` (The Background Timer)
This file uses the `node-cron` library to run tasks at specific intervals.
- **What it does:** It acts as an alarm clock for the bot.
- **Schedules:**
  - **Daily Rankings:** Runs every day at 12:00 AM IST (Indian Standard Time). It calls the `rankingBot` to calculate new damage dealt by each member since the last update and adds it to their cumulative weekly total. It also executes immediately when the server is started or restarted.
  - **Quest Auto-Starter:** Runs every 6 hours. It calls the `questStarter` to check if a quest is pending and start it if it has been waiting too long.

### `rankingBot.js` (The Leaderboard Generator)
This file is responsible for tracking and generating a cumulative weekly leaderboard.
- **What it does:** It fetches all current party members and checks their active quest participation. It compares their current pending damage against the previous snapshot to calculate new damage dealt. It persists these totals across server restarts in a local `data/damageData.json` file. At 12:00 AM IST on Sunday, it automatically resets the weekly totals back to zero to start a new week.
- **Output:** It formats this data into a simple, text-based Markdown table (excluding inactive members) showing only Rank, Player, and Weekly Damage Dealt. It then updates the Party Description and posts a notification in the party chat.

### `questStarter.js` (The Quest Manager)
This file monitors pending quests.
- **What it does:** It fetches the current party information to see if there is a pending quest. 
- **Logic:** If a quest is pending, it checks the response rate. If more than 50% of members have responded and at least one member has accepted, it will force-start the quest. If the quest is successfully started, it posts a message in the party chat.

### `welcomeBot.js` (The Greeter)
This file handles incoming webhooks from Habitica's chat.
- **What it does:** When configured, Habitica sends an HTTP POST request to your server every time a chat event occurs in your party. The `welcomeBot` parses this message.
- **Logic:** It uses a Regular Expression (`/has joined the party/i`) to check if the message is a system notification about a new member joining. If it is, it extracts the new member's username and sends a friendly welcome message in the party chat.

### `get-party-id.js` (The Setup Utility)
A simple utility script.
- **What it does:** Since Habitica no longer exposes the Party ID in the browser URL, this script uses your User ID and API Token to securely ask Habitica for your Party ID so you can put it in your `.env` file.

## 3. Exactly What Changes Are Made in Habitica?

When this bot is running, it performs the following direct actions on your Habitica account and party:

1. **Party Chat Messages:**
   - The bot will post a message saying: `🎉 Welcome to the party, @[username]! 🎉...` when a new user joins.
   - The bot will post a message saying: `📊 Daily rankings have been updated!...` every day at 12:00 AM IST and on startup.
   - The bot will post a message saying: `🚀 Quest "[questName]" has been auto-started!...` when a quest is forced to start.

2. **Party Description Updates:**
   - On server startup and every day at 12:00 AM IST, the bot will overwrite the "Party Description" field on your Habitica party page with a newly generated markdown table showing the active party members ranked by their cumulative weekly quest damage.

3. **Quest State Changes:**
   - The bot will change a quest's state from "Pending" (waiting for party members to accept/reject) to "Active" (everyone is now fighting the boss/collecting items) if the conditions are met during its 6-hour check.

## 4. Why is the Party Description Not Getting Updated?

If you have noticed that the Party Description is not updating, here are the most likely reasons:

1. **Cron Job Schedule (Timing):**
   The description is strictly programmed to update **every day at 12:00 AM IST** and **once on server startup**. If you just started the bot, it will attempt to update the description immediately. If it fails, check the server logs for API errors.

2. **Party Leader Permissions:**
   To update a party description via the Habitica API, the user account associated with the `HABITICA_USER_ID` and `HABITICA_API_KEY` **must be the Party Leader** (or have been granted explicit permissions to edit party details). If your account is just a regular member, the API request to `PUT /groups/:id` will fail due to lack of authorization.

3. **Server Sleep / Downtime:**
   If you are hosting this on a free tier service (like Render's free tier), the server will go to sleep after 15 minutes of inactivity. If the server is asleep on Sunday at 9:00 AM UTC, the Cron job will not fire, and the rankings will be skipped for that week. (To fix this, you would need to use a service like cron-job.org to ping your `/health` endpoint every 10 minutes to keep it awake).

4. **Rate Limiting or API Changes:**
   If the Habitica API is down or if the bot hit a severe rate limit, the update request might have been blocked. (Check your server logs for any `❌ Error updating weekly rankings:` messages).
